This is a mature, production-grade SMC / Order-Flow scanner codebase with:

  - 32,000+ lines of highly optimized OB / FVG / structure detection logic
    in main.py alone
  - A complete, production-proven OB backtest engine (backtest_ob.py, 847 lines)
  - A multi-exchange candle-fetch layer with time-range support ready for
    historical backtest use (get_klines_exchange_window)
  - A pure, deterministic signal-resolution engine (outcome_resolver.py)
  - Live Monitor Phase 11 (AI trading terminal, paper trading, testnet orders)
  - A comprehensive database schema covering signals, trades, monitoring, and
    a BacktestRun model that already exists

For Backtest V1 specifically:
  - BACKEND: Already ready. backtest_ob.run_ob_backtest() is production-ready,
    makes no DB writes, and is fully tested. Zero changes needed to existing
    scanner or live monitor logic.
  - CANDLE DATA: Ready. get_klines_exchange_window() supports arbitrary
    time-range fetches for historical backtesting.
  - RESOLUTION ENGINE: Ready. outcome_resolver._run_resolution_loop() is
    a pure function with 7 test cases proving correctness.
  - FRONTEND: Missing. No renderBacktest() function exists. Only a placeholder
    "This view is not yet implemented" div is rendered.
  - ROUTE: Missing. No /api/backtest endpoint exists anywhere in main.py.

Risk level for Backtest V1: LOW. All core logic is isolated and reusable.
No changes to the live scanner or live monitor are required.

FILE: backtest_ob.py (847 lines — Phases 7B through 7E + 8B)
──────────────────────────────────────────────────────────────────────────────
  Status:   COMPLETE, production-ready OB backtest engine.

  Entry Point:
    run_ob_backtest(
      limit=100, timeframe=None, pair=None, setup_type=None,
      source="live", result_filter=None, stop_mode="wick",
      freshness="all", strength_min=0.0, max_touch_count=None
    ) -> dict

  What it does:
    1. Queries SignalEvent table for OB module signals
       (setup_type in {OB_APPROACH, OB_CONSOL})
    2. Fetches pre-signal candles (1500 bars before detection) and
       post-signal candles (expiry window after detection) using
       get_klines_exchange_window()
    3. Runs 4 candle-replay variants per signal:
       - res_wick  : loss if wick crosses OB boundary
       - res_close : loss only if candle CLOSES beyond boundary (strict)
       - res_small : smaller bounce threshold (counterfactual)
       - res_cce   : candle-close entry mode (counterfactual)
    4. Classifies loss reason per signal:
       wick_stop_only, close_stop_also_lost, same_candle_stop, etc.
    5. Measures MFE/MAE (max favourable/adverse excursion from entry)
    6. Builds per-signal loss diagnostics and aggregated summaries

  Phases implemented:
    Phase 7B: Loss reason classification + counterfactual variants
    Phase 7C: Stop mode selector (wick vs. close)
    Phase 7D: First-touch freshness filter (virgin vs. already_touched)
    Phase 8B: Signal-time clearance-based touch count (no lookahead bias)

  Output structure:
    {
      "ok": bool,
      "summary": {
        checked, won, lost, expired, ambiguous, entered, errors
      },
      "freshness_summary": {

  Status:   COMPLETE, production signal-resolution engine.

  Public Functions:
    resolve_pending_signals(app, limit=50) -> dict

  Core Pure Function (KEY FOR BACKTEST):
    _run_resolution_loop(
      zone_high, zone_low, direction, bounce,
      detected_at_ms, candles, timeframe
    ) -> dict

  What the core loop does:
    1. Replays candles after detected_at_ms
    2. Detects entry (first candle wicking into zone)
    3. Tracks MFE (max favourable) and MAE (max adverse excursion)
    4. Detects WIN (target hit), LOSS (stop hit), or
       EXPIRY (EXPIRY_CANDLES limit exceeded)
    5. Handles AMBIGUOUS (target and stop hit on same candle)

  Entry logic:
    Bullish: entry when candle.low  <= zone_high → entry_price = zone_high
    Bearish: entry when candle.high >= zone_low  → entry_price = zone_low
    Target  = entry_price * (1 ± bounce%)
    Stop    = opposite zone boundary

  Expiry limits (EXPIRY_CANDLES):
    "1m":12, "3m":20, "5m":24, "15m":16, "30m":14,
    "1h":12, "2h":10, "4h":8,  "6h":6,  "12h":5,  "1d":5

  Output (per signal):
    {
      status, entry_price, entry_time, target_price, stop_price,
      exit_price, exit_time, result, result_reason,
      mfe_pct, mae_pct, time_to_entry_hours, time_to_result_hours
    }

  Test Coverage:  7 comprehensive test cases (lines ~555–676) covering
                  all resolution paths (WIN, LOSS, AMBIGUOUS, EXPIRED,
                  WAITING_FOR_ENTRY, same-candle stop, etc.)

  DB Writes:      resolve_one() writes to SignalEvent/SignalOutcome.
                  _run_resolution_loop() writes NOTHING — pure function.

  Reusable:       YES — call _run_resolution_loop() directly for backtest.

  Recommendation: Reuse _run_resolution_loop() as the backtest resolution

RISKS SPECIFIC TO BACKTEST V1
────────────────────────────────────────────────────────────────────────────────
  1. DATA DEPENDENCY
     backtest_ob.py queries SignalEvent rows. If the live scanner has
     not run for a given pair/timeframe, there will be no signals to
     backtest. This is expected behavior, not a bug. The UI should
     communicate this clearly.

  2. NO PURE HISTORICAL MODE YET
     The current backtest engine does NOT fetch historical candles and
     run detection from scratch. It replays existing SignalEvent rows.
     True "from-scratch historical backtest" (fetch 3000–4000 candles,
     detect OBs, test them) is a different architecture (Fetch-and-Detect
     mode) and is NOT present in backtest_ob.py. This is important to
     understand: Backtest V1 tests SIGNALS THAT WERE ALREADY DETECTED
     by the live scanner — it is not a raw historical scan.

     If the goal is a pure historical engine (no dependency on live
     scanner having run), a new fetch-and-detect pipeline must be built.
     backtest_ob.py cannot be reused for that purpose as-is.

  3. CANDLE FETCH LATENCY
     Fetching 1500 pre-signal + post-signal candles per signal via
     the Binance API takes time. For large limit values (limit=100+),
     the POST /api/backtest call may take 10–60 seconds. Consider
     async response or streaming for the frontend UX.

  4. RATE LIMITING
     get_klines_exchange_window() has a 12-request safety cap per call.
     For 100 signals × 2 candle windows each = 200 API calls. Binance
     allows 1200 weight/minute. Monitor carefully with large limits.

  5. LOOKAHEAD BIAS
     Phase 8B in backtest_ob.py already addresses touch-count lookahead
     bias (uses signal-time candles only, not full candle array). This
     is correct. Do not revert this behavior.
