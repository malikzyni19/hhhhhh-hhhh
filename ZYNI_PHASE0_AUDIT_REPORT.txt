B. EXISTING WATCHLIST / LIVE MONITOR LOGIC FOUND

--- NAMING CONFUSION (IMPORTANT) ---
The current codebase uses "watchlist" and "Live Monitor" as the same concept.
The desktop preview.html maps: live:"watchlist" and watchlist:"pairs" in TAB_OF_VIEW.
The mobile index.html labels the tab as "LIVE MONITOR" but backs it with id="page-watchlist".
There is NO separate "Live Monitor" model or data structure. Everything is the watchlist.

--- SERVER-SIDE WATCHLIST STORAGE ---
Location: main.py lines 255–278

Storage mechanism: /tmp/zyni_wl_{username}.json
  - Flat JSON file per user containing only a list of pair symbols (strings)
  - Example: ["BTCUSDT", "ETHUSDT", "SOLUSDT"]
  - Persists until server restart (Koyeb ephemeral storage)
  - NOT stored in the database
  - NOT associated with User model via foreign key
  - Max 10 pairs per user (enforced at register endpoint)

Functions:
  _wl_file(username) -> str          — line 262 — builds safe file path
  load_user_watchlist(username)      — line 268 — reads JSON file, returns list
  save_user_watchlist(username, pairs) — line 275 — writes JSON file

--- LIVE ORDERFLOW CACHE ---
Location: main.py lines 643–737

Global in-memory structures (shared across all users, NOT per-user):
  _wl_lock    — threading.Lock()
  _wl_pairs   — List[str], current active watchlist pairs
  _wl_cache   — Dict[str, Any], symbol → latest orderflow result
  _wl_thread  — Optional[threading.Thread], background polling thread

Background thread:
  _ensure_wl_thread()         — starts thread if not running
  _wl_background_loop()       — continuous orderflow polling loop

WebSocket management:
  start_ob_ws(symbol)         — starts orderbook WebSocket for a pair
  stop_ob_ws(symbol)          — stops WebSocket for a removed pair

CRITICAL RISK: _wl_pairs and _wl_cache are GLOBAL (single shared state).
  If two users have different watchlists, they will overwrite each other.
  The last user to call /api/watchlist/register wins the global state.

E. EXISTING SCAN DATA FIELDS AVAILABLE FOR SETUP SNAPSHOT
================================================================================

--- FROM /api/scan RESPONSE (per alert row) ---
Each scan alert row contains at minimum:
  symbol          — trading pair (e.g. "BTCUSDT")
  exchange        — "binance", "bybit", "okx", "mexc"
  market          — "perpetual", "spot"
  timeframe       — "15m", "30m", "1h", "4h", "1d"
  direction       — "bullish" or "bearish"
  score           — integer confidence score (0-100)
  zone_high       — float, top of OB/FVG/FIB zone
  zone_low        — float, bottom of OB/FVG/FIB zone
  current_price   — float, price at scan time
  setup_type      — "OB_APPROACH", "OB_CONSOL", "FVG_FILL", etc.
  topAlert        — object with type and meta

--- FROM topAlert.meta (OB-specific fields) ---
  obQuality       — integer 0-100, OB quality score
  obAgeBars       — integer, how many bars since OB formed
  obBar           — integer, bar index of OB formation
  isVirginOb      — boolean, whether OB has been tested
  tvObVolumeSharePct    — float, TradingView OB volume share
  tvObFormationVolume   — float, OB formation candle volume
  tvObVisibleCount      — integer, visible OB count at time of scan
  touches         — integer, number of OB touches

--- FROM topAlert.meta (FVG-specific fields) ---
  fvg_level       — float, FVG midpoint
  fvg_state       — "open", "closed", "partially_filled"
  fvg_age         — integer bars
  fvg_is_valid    — boolean
  fvg_is_bag      — boolean (bag FVG)
  fvg_touch_depth — "none", "touched", "half", "full"

--- FROM topAlert.meta (FIB-specific fields) ---
  fib_level       — string e.g. "0.618", "0.786"
  fib_tf          — timeframe for leg detection
  fib_leg_high    — float
  fib_leg_low     — float
  fib_progression — float, how far price has retraced

--- FROM analyze_pair() return (available for watchlist refresh) ---
  rsi             — float, RSI value
  atr             — float, ATR value
  volume          — float, current bar volume
  bias            — "bullish", "bearish", "neutral"
  structure       — trend/structure state

--- FROM /api/watchlist/refresh RESPONSE (per pair result) ---
  symbol
  tfs: {
    "15m": { score, direction, topAlert, zone_high, zone_low, ... },
    "1h":  { score, direction, topAlert, zone_high, zone_low, ... },
    "4h":  { score, direction, topAlert, zone_high, zone_low, ... }
  }

--- FROM SignalEvent MODEL (db — already saved fields for signals) ---
  signal_id, pair, module, timeframe, direction, score,
  zone_high, zone_low, detected_price, detected_at, exchange,
  setup_type, raw_setup, raw_meta_json (JSON blob with full meta)

--- SOURCE TAB FIELD ---
  NOT currently tracked in scan response.
  The source tab (Main Scan vs Compressed vs Bias Shift etc.) is known
  client-side at click time but is NOT included in server scan results.
  Must be added as a field when saving the LiveMonitorItem.

--- MISSING FIELDS FOR FULL SETUP SNAPSHOT ---
  source_tab      — which scanner tab produced this setup (client-side only)
  added_at        — timestamp when user added to Live Monitor
  user_id         — for per-user DB persistence (currently only /tmp files)
  health_score    — computed Live Monitor health metric (does not exist yet)
  notes           — user annotation (does not exist yet)

  "watchlist" tab is in ALL_TABS (models.py line 9)

  This means Live Monitor tab may be locked for standard users unless admin overrides.

I. RISKS / DO-NOT-BREAK AREAS
================================================================================

RISK 1: GLOBAL WATCHLIST CACHE CONFLICT
  The _wl_pairs and _wl_cache globals (main.py ~line 643) are shared across ALL users.
  When User A registers their watchlist, it overwrites User B's active pairs.
  The new LiveMonitorItem system MUST use per-user DB storage, not these globals.
  DO NOT extend the global cache for per-user Live Monitor.
  The existing /api/watchlist/* endpoints can remain as-is for legacy use,
  but new /api/live-monitor/* must use DB, not the global cache.

RISK 2: db.create_all() vs EXISTING TABLES
  db.create_all() is safe for NEW tables — it will create LiveMonitorItem and
  LiveMonitorEvent without touching existing tables.
  DO NOT add columns to existing tables using db.create_all() alone —
  it will NOT apply ALTER TABLE changes to already-existing tables.
  For any additions to existing models (if needed), add idempotent SQL in the
  existing migration block already in main.py (lines 81–103+).

RISK 3: MOBILE UI BREAK RISK
  The mobile index.html is a completely separate UI with its own watchlist logic.
  Any changes to /api/watchlist/* endpoint contracts will break mobile.
  Solution: New /api/live-monitor/* endpoints are SEPARATE from existing
  /api/watchlist/* endpoints. Keep watchlist endpoints untouched.
  Mobile can have a separate "Add to LM" button/flow calling new endpoints.

RISK 4: DUPLICATE JS ID RISK
  preview.html uses inline JS with many global variable names and element IDs.
  Adding new Live Monitor HTML sections could conflict with existing IDs.
  Check: page-watchlist, btnLM, wlRefreshBtn, etc. before adding new elements.
  New LM items panel must use unique IDs (e.g., lm-items-list, lm-selected-panel).

RISK 5: "Add to Live Monitor" MISSING FULL SETUP DATA
  Current btnLM click handler only captures the symbol.
  The full setup snapshot (zone, direction, OB meta) is displayed in the insight
  panel at click time but NOT serialized or sent to server.
  Risk: If insight panel state is stale or symbol was de-selected, snapshot is wrong.
  Solution: At click time, serialize the currently displayed insight panel's full data
  object (the same JS object used to render the panel) into the POST body.

RISK 6: ENDPOINT NAMING CONFLICT
  /api/watchlist/* endpoints all exist and are used by both desktop and mobile.
  New endpoints MUST use /api/live-monitor/* prefix to avoid collisions.
  DO NOT rename or repurpose any /api/watchlist/* endpoint.

RISK 7: TOKEN / PERMISSION SIDE EFFECTS
  "watchlist" tab is NOT in default "user" role allowed_tabs.
  If Live Monitor becomes a new tab (or stays as "watchlist" tab), standard users
  may not have access without admin granting permission.
  Decision needed: Should Live Monitor use the existing "watchlist" tab permission
  or a new "live_monitor" tab permission?
  Either way, do not change permission checks for existing scan tabs.

RISK 8: LOCALSTORAGE CONFLICT
  Mobile watchlist is partially saved to localStorage keys:
    "quickWatchlist" — array of symbols
    "zyni_watchlists" — named watchlist map
  New LiveMonitorItem DB system must not conflict with these keys.
  Keep localStorage as a client-side cache/fallback only — source of truth is DB.

RISK 9: /tmp FILE STORAGE OBSOLESCENCE
  load_user_watchlist() and save_user_watchlist() use /tmp files.
  These will be obsolete once DB-backed LiveMonitorItem is live.
  Do NOT delete them in Phase 1 — they are still used by /api/watchlist/register
  and /api/watchlist/get which support the existing watchlist flow.
  Only retire them when the entire watchlist system is migrated.

RISK 10: DEEPSEEK CHATBOX INTEGRATION
  No DeepSeek API key, client library, or endpoint exists in the repo.
  The hero section chatbox requires a new backend proxy endpoint and API key.
  This is a new dependency with cost implications (API calls per message).
  Risk: Rate limiting, key exposure in frontend, context size management.
  The chatbox must be scope-limited to the selected setup context only.

RISK 11: LOCKED UI REFERENCE FILE NOT IN REPO
  zyni_live_monitor_preview.html is referenced as the locked UI direction
  but was NOT found in the repo. Phase 1 frontend work cannot begin without it.
  Must be added to repo or shared separately before frontend implementation.

RISK 12: MONOLITHIC TEMPLATE FILE SIZE
  preview.html is extremely large (thousands of lines, all inline).
  Adding Live Monitor right-panel HTML and JS sections increases this further.
  Risk: Browser parse time, accidental JS scope collisions, hard to debug.
  Recommendation: Use template {% block %} or at minimum add clear section comments.

  GET  /api/live-monitor/items
    - @login_required
    - username = session.get("username")
    - user = _DBUser.query.filter_by(username=username).first()
    - items = LiveMonitorItem.query.filter_by(user_id=user.id).order_by(added_at.desc()).all()
    - Return JSON array of items

  POST /api/live-monitor/items
    - @login_required
    - Parse JSON body: symbol, exchange, market, source_tab, setup_type,
      direction, timeframe, zone_high, zone_low, confidence, score,
      current_price, snapshot_json
    - Validate required fields
    - Create LiveMonitorItem, db.session.add(), db.session.commit()
    - Return new item as JSON

    - @login_required
    - Fetch item by id
    - Verify item.user_id == current user id (ownership check)
    - db.session.delete(item), db.session.commit()
    - Return {"deleted": item_id}

  GET /api/live-monitor/events/<int:item_id>
    - @login_required
    - Verify item ownership
    - events = LiveMonitorEvent.query.filter_by(item_id=item_id).order_by(created_at.desc()).all()
    - Return JSON array
