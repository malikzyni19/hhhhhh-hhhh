--- GET /api/live-monitor/items ---
  Location: main.py line 8017
  Auth:     @login_required
  Query param: ?active_only=1 (optional — filters to is_active=True only)
  Returns:  { "items": [...], "count": N }
  Each item: full _live_monitor_item_to_dict() output including parsed snapshot.

--- POST /api/live-monitor/items ---
  Location: main.py line 8033
  Auth:     @login_required
  Body:     JSON with symbol (required) + optional snapshot fields
  Logic:
    - Normalizes symbol to uppercase
    - Defaults: exchange="binance", market="perpetual", source_tab="unknown", status="watching"
    - Merges topAlert + meta into snapshot_json blob
    - Duplicate check: same user + symbol + setup_type + timeframe + zone → upserts existing
    - Fallback duplicate: same user + symbol + setup_type + timeframe (no exact zone)
    - Creates LiveMonitorEvent (type="added" or "updated") with flush/commit
    - Returns: { "ok": true, "item": {...}, "created": true/false }

--- DELETE /api/live-monitor/items/<int:item_id> ---
  Location: main.py line 8158
  Auth:     @login_required
  Logic:
    - Ownership verified: row.user_id must match current user — 403 if not
    - Soft delete: is_active=False, status="removed"
    - Creates LiveMonitorEvent (type="removed")
    - Returns: { "ok": true, "deleted": item_id }

--- GET /api/live-monitor/events/<int:item_id> ---
  Location: main.py line 8193
  Auth:     @login_required
  Logic:
    - Ownership verified on the parent item
    - Returns events newest-first
    - Returns: { "events": [...], "count": N }

5. HOW ADD TO LIVE MONITOR NOW SAVES FULL SNAPSHOT
================================================================================

Before Phase 1:
  User clicks "Add to Live Monitor" →
    symbol → ST.stars → syncWatchlist() → POST /api/watchlist/register
    Only the pair symbol is saved. No zone, no direction, no timeframe, no meta.
    Stored in /tmp/zyni_wl_{username}.json (ephemeral, server-side text file).

After Phase 1:
  User clicks "Add to Live Monitor" →
    [1] symbol → ST.stars → syncWatchlist()  ← LEGACY STILL WORKS
    [2] Full snapshot built from insight panel row object:
          symbol, exchange, market, source_tab (which scanner tab),
          setup_type (OB_APPROACH / FVG / FIB / etc.),
          direction (bullish/bearish),
          timeframe (15m/1h/4h/etc.),
          zone_high, zone_low (from zoneOf() helper — exact OB/FVG/FIB zone),
          confidence/score,
          current_price,
          topAlert (full object),
          meta (full meta object with obTop/obBottom/fvgTop/etc.),
          snapshot.row (full result row),
          selected_timeframes: ["15m","1h","4h"],
          selected_modules: ["OB","FVG","FIB","Bias"]
        → POSTed to /api/live-monitor/items
    [3] Backend saves to DB as LiveMonitorItem row (per-user, persistent)
    [4] LiveMonitorEvent created (type="added") with source_tab + setup_type + timeframe
    [5] DB item persists through page refresh and server restart
    [6] liveMonitorItems JS state updated via loadLiveMonitorItems()

7. ISSUES / RISKS FOUND DURING IMPLEMENTATION
================================================================================

ISSUE 1: GLOBAL CACHE STILL SHARED (pre-existing, not introduced by Phase 1)
  _wl_pairs and _wl_cache are still global in main.py. The new LiveMonitorItem
  system is fully DB-backed and per-user — it does NOT use these globals.
  No action needed in Phase 1. This remains a Phase 2+ concern if the old
  watchlist polling is kept alive alongside the new system.

ISSUE 2: "watchlist" TAB PERMISSION FOR STANDARD USERS
  Regular "user" role does not include "watchlist" in allowed_tabs by default.
  The Live Monitor tab (data-view="live") routes to v==="watchlist" in the
  permission check. Standard users may hit the lock popup.
  RECOMMENDATION FOR PHASE 2: Either add "watchlist" to user role defaults
  in permissions.py, or add a new "live_monitor" tab permission.
  This was pre-existing — Phase 1 did not change permissions.

ISSUE 3: SNAPSHOT SIZE
  The snapshot payload includes the full row object (r) which may include
  sparkline arrays. For large sparklines this could exceed a few KB per item.
  The TEXT column type in PostgreSQL has no limit so this is not a hard failure,
  but could be worth trimming sparkline data from snapshot in Phase 2.

ISSUE 4: LOCKED UI FILE STILL NOT IN REPO
  zyni_live_monitor_preview.html is referenced as the locked UI direction
  but is not in the repository. Phase 2 UI wiring will need this file.

NO REGRESSIONS INTRODUCED:
  - Mobile UI (index.html) was not modified
  - No existing scan tab logic was changed
  - No existing watchlist endpoint was modified
  - No existing model was changed
