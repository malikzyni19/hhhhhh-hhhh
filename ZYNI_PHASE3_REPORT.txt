2. BACKEND HELPERS ADDED (main.py)
================================================================================

  All four helpers inserted between existing _lm_int_or_zero and
  _live_monitor_item_to_dict, in the Live Monitor backend section.


  2a. _LM_VALID_TIMEFRAMES / _LM_VALID_MODULES (module-level constants)
  ──────────────────────────────────────────────────────────────────────
    _LM_VALID_TIMEFRAMES = ["15m", "30m", "1h", "4h", "1d"]
    _LM_VALID_MODULES    = ["OB", "FVG", "FIB", "Breaker", "Bias"]


  2b. _lm_allowed_timeframes(value) -> list
  ─────────────────────────────────────────
    - Accepts: stored JSON string, list, or None
    - Parses JSON string safely (no crash on bad JSON)
    - Normalizes to lowercase
    - Removes "5m" if present (hidden execution TF — never scanned)
    - Removes any TF not in _LM_VALID_TIMEFRAMES
    - Returns ["15m","30m","1h","4h","1d"] on empty/invalid input


  2c. _lm_allowed_modules(value) -> list
  ────────────────────────────────────────
    - Accepts: stored JSON string, list, or None
    - Parses JSON safely
    - Normalizes via alias dict:
        "ob" → "OB", "fvg" → "FVG", "fib" → "FIB"
        "breaker"/"breakerblock" → "Breaker", "bias" → "Bias"
    - Deduplicates while preserving order
    - Returns ["OB","FVG","FIB","Breaker","Bias"] on empty/invalid input


  2d. _lm_build_scan_config(item) -> dict
  ─────────────────────────────────────────
    - Calls _lm_allowed_timeframes and _lm_allowed_modules
    - Builds a wl_config dict fully compatible with existing _scan_pair_multitf
    - Per-TF OB approach distances:
        15m → 0.8%, 30m → 1.0%, 1h → 1.5%, 4h → 2.5%, 1d → 3.0%
    - Per-TF FVG age max:
        15m → 5, 30m → 8, 1h → 10, 4h → 15, 1d → 20
    - fib_tf selected from [1h, 4h, 30m, 1d, 15m] — picks first present in tfs
    - scan_ob/fvg/fib/breaker/bias_1d set from module list
    - 5m is never present in timeframes (filtered by _lm_allowed_timeframes)


  2e. _lm_extract_mtf_summary(scan_result, tfs, mods) -> dict
  ─────────────────────────────────────────────────────────────
    - Converts the raw _scan_pair_multitf result dict to a clean UI/storage summary
    - Per-TF processing:
        trend/itrend → direction label (bullish/bearish/neutral)
        OB:      inside → "strong/In Zone", approaching (quality≥70) → "strong/Approach",
                 approaching (quality<70) → "yes/Approach", none → "none/—"
        FVG:     untouched+valid → "yes/Untouched", untouched → "warn/Touched",
                 none → "none/—"
        FIB:     best by |dist| → "yes/<level>", none → "none/—"
        Breaker: inside → "strong/In Zone", approaching+highProb → "yes/Approach",
                 approaching → "warn/Approach", none → "none/—"
        Bias:    per-TF trend (1d TF uses bias_1d override):
                 +1 → "strong/Bull", -1 → "strong/Bear", 0 → "none/—"
    - TF with error → {"error": "<msg>"}
    - Returns:
      {
        "symbol", "market", "exchange",
        "timeframes": [...],  "modules": [...],
        "tfs": {
          "<tf>": {
            "score", "direction", "price",
            "modules": {"OB":{"state","label"}, "FVG":{...}, ...}
          }
        },
        "refreshed_at": ISO,
        "phase": "phase3_mtf_scan"
      }
    - Never fakes strong values — "none/—" is the safe fallback

3. NEW ENDPOINTS ADDED (main.py)
================================================================================

  3a. POST /api/live-monitor/items/<int:item_id>/refresh
  ───────────────────────────────────────────────────────
    @login_required

    Checks:
      - uid required (401)
      - item must exist (404)
      - item.user_id == uid (403)
      - item.is_active must be True (400 "inactive")

    Flow:
      1. _lm_allowed_timeframes / _lm_allowed_modules / _lm_build_scan_config
      2. _scan_pair_multitf(symbol, market, config, exchange)  ← REUSED directly
      3. _lm_extract_mtf_summary(result, tfs, mods)
      4. _json_loads_safe(row.snapshot_json) → merge:
             snap["latest_mtf_scan"]     = mtf_summary
             snap["last_mtf_refresh_at"] = ISO timestamp
             snap["timeframe_policy"]    = preserved / added if missing
      5. row.current_price = scan_result["price"] if > 0
      6. row.snapshot_json = _json_dumps_safe(snap)
      7. row.updated_at = now
      8. LiveMonitorEvent inserted:
             event_type = "mtf_refreshed"
             event_description = "MTF scan refreshed"
             details_json = {timeframes, modules, phase}
      9. db.session.commit()

    Returns: {"ok": true, "item": {...}, "latest_mtf_scan": {...}}
    Errors:  {"error": "not_found"|"forbidden"|"inactive"|"scan_failed"|"db"}


  3b. POST /api/live-monitor/refresh-all
  ────────────────────────────────────────
    @login_required

    Checks: uid required (401)

    Flow:
      1. Load active items for user, newest first, limit 10 (_MAX_ITEMS = 10)
      2. For each item: same scan/merge flow as individual refresh
         - try/except per item: one failure does not abort others
         - errors collected in errors[] list
      3. db.session.commit() for all successful updates in one transaction
      4. Re-query refreshed rows, serialize and return

    No event logged per item (avoids event spam on bulk refresh).

    Returns:
      {
        "ok": true,
        "updated": N,
        "failed": M,
        "items": [...serialized active items...],
        "errors": [{"symbol":"...", "error":"..."}]
      }

4. HOW _scan_pair_multitf WAS REUSED
================================================================================

  _scan_pair_multitf already accepts a wl_config dict with:
    timeframes, scan_ob, scan_fvg, scan_fib, scan_breaker, bias_1d,
    ob_approach, fvg_age_min, fvg_age_max,
    fib_tf, fib_tolerance, fib_approach, fib_atr_mult, fib_levels,
    breaker_approach_pct, breaker_max_age, breaker_require_fvg

  _lm_build_scan_config() produces exactly this structure from a LiveMonitorItem.
  _scan_pair_multitf is called directly — no duplication of scan logic.

  The function is called once per symbol (not the full market).
  5m is never in the timeframes list, so it is never scanned.

5. HOW latest_mtf_scan IS STORED
================================================================================

  LiveMonitorItem.snapshot_json (existing TEXT column) stores a JSON object.
  On refresh, the existing snapshot is preserved and two keys are merged in:

    snapshot["latest_mtf_scan"]     = <_lm_extract_mtf_summary output>
    snapshot["last_mtf_refresh_at"] = ISO timestamp
    snapshot["timeframe_policy"]    = preserved if already present, added if missing

  The DB schema is NOT changed — no migration needed.

  _live_monitor_item_to_dict serializes snapshot_json as item.snapshot (parsed dict).
  On the frontend, item.snapshot.latest_mtf_scan is therefore directly accessible.

6. HOW MTF MATRIX NOW READS REAL SCAN DATA
================================================================================

  _lmMtfMatrix(item) updated (templates/preview.html):

  Priority:
    1. item.snapshot.latest_mtf_scan exists AND scan.tfs[tf] exists:
       → reads scan.tfs[tf].modules[mod].{state, label} for each cell
       → state maps to .mc class: strong / yes / warn / none
       → Bias uses green/red color for Bull/Bear labels

    2. scan.tfs[tf].error:

    3. No scan data (latest_mtf_scan absent or TF not in scan):
       → falls back to snapshot-based placeholder (existing Phase 2 logic)

  Rows (5): OB, FVG, FIB, Breaker, Bias
  Columns:  item.selected_timeframes (default ["15m","30m","1h","4h","1d"])
  No 5m column — 5m is never in selected_timeframes.

  Footer:
    - Live scan: "Live MTF scan · Last refreshed X ago"  (using _lmAgo)
    - No scan:   "Snapshot-based · Refresh to run Phase 3 MTF scan"

11. ISSUES AND RISKS FOUND
================================================================================

  ISSUE 1 — FVG distance not available
    FVGs in scan_result do not carry a dist field (unlike OBs, FIBs, Breakers).
    _lm_extract_mtf_summary uses untouched/touched status as the signal instead.
    No workaround needed — status is sufficient for the matrix display.

  ISSUE 2 — Price per TF
    _scan_pair_multitf stores price from the 1h TF close (or first successful TF).
    All per-TF rows in the summary share the same price value.
    This is consistent with existing scanner behavior.

  RISK 1 — Scan latency
    _scan_pair_multitf fetches klines from exchange for each TF.
    With 5 TFs × up to 10 items in refresh-all, this is sequential and can be slow.
    Phase 4+ can parallelize refresh-all with ThreadPoolExecutor.
    For Phase 3, sequential is safe and correct.

  RISK 2 — Token budget not consumed
    Unlike /api/watchlist/refresh, the new LM refresh endpoints do not call
    consume_tokens(). This is intentional for Phase 3 — the token system
    targets the old watchlist flow. Phase 4 can add budgeting if needed.

  RISK 3 — /api/watchlist/cache polling still runs
    startLiveMonitorPolling() still polls /api/watchlist/cache every 5s
    (legacy order-flow cache). This data is stored in _lmCache but is no longer
    used in the LM UI after Phase 2. It can be removed in a future cleanup phase.
