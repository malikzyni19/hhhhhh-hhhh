2. FIX 1 — HEALTH SCORE RECURSION PREVENTION
================================================================================

  Problem:
    _lm_compute_health() used item.score as raw_score (base_pts input).
    After refresh, row.score = health["health_score"].
    On the next refresh, item.score was already the previous health score,
    creating a self-reinforcing feedback loop in the base_pts component.

  Solution:
    New helper: _lm_original_setup_score(item, snapshot) -> int

    Returns the original signal/setup confidence score by searching these
    sources in priority order (first non-None result in 0–100 range wins):

      1. snapshot["original_setup_score"]         — preserved value
      2. snapshot["row"]["score"]                 — watchlist row score
      3. snapshot["row"]["confidence"]            — watchlist row confidence
      4. snapshot["topAlert"]["score"]            — top alert score
      5. snapshot["topAlert"]["confidence"]       — top alert confidence
      6. item.confidence                          — never overwritten by engine
      7. item.score  only if snapshot has NO latest_health yet
                         (first-run safety net; harmless before first write)
      8. 0                                        — absolute fallback

    Uses safe float→int conversion; ignores "", None, "N/A", out-of-range.

  In _lm_compute_health():
    BEFORE:  raw_score = item.score or 0
    AFTER:   raw_score = _lm_original_setup_score(item, snap)

  In all 3 endpoints (before _lm_compute_health call):
    if "original_setup_score" not in snap:
        snap["original_setup_score"] = _lm_original_setup_score(row, snap)

    This runs once (guarded), storing the original score in snapshot_json
    so future recalculations always find it at priority 1 and never fall
    through to the now-polluted item.score.

  Endpoints updated:
    - POST /api/live-monitor/items/<id>/refresh     ✓
    - POST /api/live-monitor/refresh-all            ✓  (inside the for loop)
    - POST /api/live-monitor/items/<id>/recalc-health ✓

  Checklist label:
    BEFORE: "Score ≥ 35"
    AFTER:  "Original Score ≥ 35"

  Verification:
    [PASS] _lm_original_setup_score defined
    [PASS] _lm_compute_health calls _lm_original_setup_score
    [PASS] item.score not used directly as raw_score in _lm_compute_health
    [PASS] original_setup_score preservation in all 3 endpoints

3. WHERE original_setup_score IS STORED
================================================================================

  Location:   snapshot_json TEXT column → parsed as snap dict
  Key:        snap["original_setup_score"]   (integer 0–100)

  Written:    First time any of the 3 health endpoints runs for an item.
  Guarded:    if "original_setup_score" not in snap — never overwritten.
  Read:       _lm_original_setup_score() checks this at priority 1.

  This means:
    - Existing items with previous health will get their original score
      captured on the very next refresh or recalc.
    - New items get it on first refresh.
    - Once stored, it never changes — even if item.score changes later.

4. FIX 2 — ZONE HIGH / LOW NORMALIZATION
================================================================================

  Problem:
    _lm_zone_health() used item.zone_high / item.zone_low directly,
    assuming zone_high >= zone_low. Reversed values broke inside/breach logic.

  Solution:
    Inside _lm_zone_health():

      try:
          raw_zh = float(item.zone_high)
          raw_zl = float(item.zone_low)
      except (TypeError, ValueError):
          return {... "reason": "No zone set"}

      if raw_zh <= 0 or raw_zl <= 0:
          return {... "reason": "No zone set"}

      zh = max(raw_zh, raw_zl)   # normalized zone_high
      zl = min(raw_zh, raw_zl)   # normalized zone_low

    All inside/near/breach calculations then use zh and zl.

  Return values:
    latest_health.zone.zone_high = zh (normalized)
    latest_health.zone.zone_low  = zl (normalized)

  Frontend receives normalized values — no UI change required.

5. FIX 3 — BREACH RISK THRESHOLD
================================================================================

  Problem:
    Any price below zone_low (bullish) or above zone_high (bearish) — even
    a 0.01% tick — immediately flagged breach_risk. Too aggressive for
    crypto zones where price can wick just outside the zone briefly.

  New constants (added at module level near _LM_VALID_*):

    _LM_ZONE_NEAR_PCT   = 2.0    # % outside zone edge = "near"
    _LM_BREACH_RISK_PCT = 0.25   # % against-zone required for breach_risk

  Updated logic in _lm_zone_health():

    BULLISH:
      price > zh  → distance from high → "near" (≤2%) or "watching" (>2%)
                    breach_risk = False in this branch
      price < zl  → distance from low (= (zl - price) / zl * 100)
                    if distance > 0.25% → breach_risk = True, status = "breach_risk"
                    if distance ≤ 0.25% → breach_risk = False, status = "near"

    BEARISH:
      price < zl  → distance from low → "near" (≤2%) or "watching" (>2%)
                    breach_risk = False in this branch
      price > zh  → distance from high (= (price - zh) / zh * 100)
                    if distance > 0.25% → breach_risk = True, status = "breach_risk"
                    if distance ≤ 0.25% → breach_risk = False, status = "near"

    NEUTRAL:
      No breach risk flag. inside / near / watching only.

  Distance measurement:
    Uses nearest zone edge (zl or zh), not midpoint.
    This exactly matches the spec examples.

  Spec examples verified:
    Bullish zone zl=100, zh=105:
      price 99.90 → dist = (100-99.90)/100 = 0.10% → ≤0.25% → "near"   ✓
      price 99.60 → dist = (100-99.60)/100 = 0.40% → >0.25% → "breach_risk" ✓

    Bearish zone zl=100, zh=105:
      price 105.10 → dist = (105.10-105)/105 ≈ 0.095% → ≤0.25% → "near" ✓
      price 105.50 → dist = (105.50-105)/105 ≈ 0.476% → >0.25% → "breach_risk" ✓

  Reason string:
    BEFORE: "Zone breached"
    AFTER:  "Zone breach risk"

  Near threshold unchanged:
    _LM_ZONE_NEAR_PCT = 2.0  — same as before, no change to near logic.

    [PASS] Both constants used inside _lm_zone_health
