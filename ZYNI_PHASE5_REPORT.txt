    _lm_market_context_default(symbol, exchange, market, reason, provider) -> dict
      Safe placeholder. Returns all 6 context sections with available=False.
      Used when provider fails or exchange unsupported.

    _lm_fetch_market_context(symbol, exchange, market) -> dict
      Router function. Normalizes exchange/market to lowercase.
      Routes to Binance provider if exchange=="binance".
      Returns default with reason if exchange unsupported.
      Catches all exceptions, never raises.

    _lm_fetch_binance_market_context(symbol, market) -> dict
      Binance implementation. Calls 5 public endpoints.
      Computes funding/L-S/taker biases. Builds summary.

    _lm_attach_market_context(row, snapshot, force) -> tuple
      Fetches and stores market context into snapshot dict.
      Does NOT commit. Does NOT modify health_score.
      Returns (snapshot, market_ctx).

3. BINANCE PUBLIC ENDPOINTS USED
================================================================================

  All endpoints are public — no API key required:

  1. GET /fapi/v1/premiumIndex?symbol=SYMBOL
     Extracts: lastFundingRate, nextFundingTime, markPrice
     Used for: funding rate + mark price for OI USD calc

  2. GET /fapi/v1/openInterest?symbol=SYMBOL
     Extracts: openInterest
     Used for: open interest token quantity

  3. GET /fapi/v1/ticker/24hr?symbol=SYMBOL
     Extracts: quoteVolume, priceChangePercent
     Used for: activity tile (24h volume + price change)

  4. GET /futures/data/globalLongShortAccountRatio?symbol=SYMBOL&period=15m&limit=1
     Extracts: longShortRatio, longAccount, shortAccount
     Used for: long/short ratio tile

  5. GET /futures/data/takerlongshortRatio?symbol=SYMBOL&period=15m&limit=1
     Extracts: buySellRatio, buyVol, sellVol
     Used for: taker pressure tile

  Timeout: 5 seconds per request.
  Liquidations: NOT fetched from REST (real-time liquidation stream is
  WebSocket-only). Liquidation tile shows "Awaiting liquidation feed".

4. NEW MARKET CONTEXT ENDPOINTS ADDED
================================================================================

  POST /api/live-monitor/items/<id>/refresh-market-context
    - @login_required
    - Ownership check (user_id)
    - Active item check
    - Calls _lm_attach_market_context()
    - No MTF scan, no health recompute, no event created
    - Returns {ok, item, market_context}

  POST /api/live-monitor/refresh-market-context-all
    - @login_required
    - Current user's active items only, max 10
    - Fetches market context for each item
    - Continues if one item fails
    - Single DB commit after loop
    - Returns {ok, updated, failed, items, errors}

  Total Live Monitor endpoints: 11
    (3 Phase 1/2 CRUD + 2 Phase 3 MTF refresh + 1 Phase 4 recalc-health
     + 2 Phase 4.5 session + 2 Phase 5 market context)

6. HOW UNSUPPORTED EXCHANGES ARE HANDLED
================================================================================

  _lm_fetch_market_context() normalizes exchange to lowercase.
  If exchange != "binance":
    Returns _lm_market_context_default(..., reason="Unsupported exchange for Phase 5")
    All tiles show "Unavailable" / "Awaiting"

  All Bybit/OKX/MEXC items will show safe placeholder until providers added.
  No crash, no partial fake data.


================================================================================
7. HOW PARTIAL FAILURES ARE HANDLED
================================================================================

  _lm_fetch_binance_market_context():
    - Each of the 5 endpoints is wrapped in its own try/except.
    - If one endpoint fails, its section stays as the default (available=False).
    - Successfully fetched sections are still returned.
    - errors[] list tracks which endpoints failed.
    - ctx["ok"] = True if at least funding or taker_pressure was fetched.
    - ctx["ok"] = False if all critical endpoints fail.
    - No exception escapes the provider function.

  _lm_attach_market_context():
    - Wraps _lm_fetch_market_context in try/except.
    - On total failure, stores default context with reason.
    - Never crashes the calling endpoint.

8. MARKET CONTEXT AND MTF REFRESH — OPTION A (SEPARATE)
================================================================================

  Chosen: Option A — Market context is NOT fetched during MTF Refresh.

  Reason:
    - Avoids extra Binance calls on every Refresh Setup click.
    - Avoids slowing down Refresh All (which is already doing 5-TF scans).
    - Phase 5 context is display-only, not time-critical.
    - User clicks "Market" button in card header when they want fresh data.

  MTF refresh endpoints (api_lm_items_refresh, api_lm_refresh_all):
    - Unchanged. No market context calls added.
    - Phase 4.5 session context still fetched in these (pure Python, instant).

  Market context endpoints are standalone:
    - refresh-market-context (per item)
    - refresh-market-context-all (batch)

14. RISKS / ISSUES FOUND
================================================================================

  Geo-blocking:
    fapi.binance.com may be blocked in some hosting regions (HTTP 451).
    The existing codebase already documents this (BINANCE_FUTURES_API comment
    mentions geo-blocking). If all 5 endpoints return 451 or fail, the tiles
    will show "Awaiting" — no crash, graceful degradation.

  Rate limiting:
    Public Binance endpoints are rate-limited. With Option A (separate refresh),
    market context is only fetched when the user clicks "Market". No automatic
    polling. Risk is low.

  Liquidation data:
    Real-time liquidation data is not available via Binance public REST.
    The WebSocket stream (aggTrade for liquidations) is not implemented.
    This is documented: liquidations.available = False. Future provider can
    add this when a reliable REST-based source is available.

  OI USD calculation:
    OI in USD is computed as: openInterest * markPrice.
    Both are fetched from the same premiumIndex endpoint, so the mark price
    may be slightly stale if the second call is slightly delayed.
    This is acceptable for display purposes.
