2. ISSUE 1 — CONSERVATIVE CONSENSUS TIE LOGIC
================================================================================

  Problem:
    Original tie-break used aggressive weights (confirmed_watch = 5 wins ties).
    A tie between "watch" and "high_risk" would return "watch" — unsafe.

  Fix:
    Replaced _VERDICT_WEIGHT max() with conservative rank min():

    Conservative rank (lower = more conservative, wins ties):
      avoid           = 0
      high_risk       = 1
      wait            = 2
      watch           = 3
      confirmed_watch = 4

    Tie-break logic:
      max_votes = max(verdict_votes.values())
      tied_verdicts = [v for v, cnt in verdict_votes.items() if cnt == max_votes]
      winner_verdict = min(tied_verdicts, key=lambda v: _CONSERVATIVE_RANK.get(v, 2))

    Positive/risk split detection:
      If both positive_votes > 0 AND risk_votes > 0:
        disagreement = True
        disagreement_note = "Agents split between positive and risk verdicts..."
        If winner is on positive side (watch or confirmed_watch):
          If risk_votes >= positive_votes: winner = "high_risk"
          Else:                           winner = "wait"

    General disagreement (no positive/risk split but low agreement):
      If unique_verdicts > 1 AND agreement < 0.67:
        disagreement = True
        disagreement_note = "Agents disagree on this setup."

    New fields added to consensus return:
      "disagreement": true/false
      "disagreement_note": ""

  Verification (8/8 test cases):
    [PASS] watch vs high_risk (tie) -> high_risk
    [PASS] confirmed_watch vs wait (tie) -> wait
    [PASS] confirmed_watch vs avoid (tie) -> avoid
    [PASS] confirmed_watch x2 vs high_risk x1 -> wait (split, positive > risk)
    [PASS] confirmed_watch x1 vs high_risk x2 -> high_risk (split, risk >= positive)
    [PASS] watch x2 vs avoid x1 -> wait (split, positive > risk)
    [PASS] confirmed_watch x3 (unanimous) -> confirmed_watch (no downgrade)
    [PASS] wait x2 vs watch x1 -> wait (majority, no split)

4. ISSUE 3 — AGENT-SPECIFIC COOLDOWN IN ai-analyze
================================================================================

  Problem:
    Cooldown checked only time, not which agent produced the cached result.
    Switching from DeepSeek to Claude within 10s returned DeepSeek cache.

  Fix:
    Resolve requested agent BEFORE cooldown check:
      resolved_agent    = _lm_get_ai_agent_config(agent_id)
      resolved_agent_id = resolved_agent.get("id", "local_fallback")

    Cooldown only triggers if BOTH conditions met:
      1. diff < 10 seconds
      2. cached_agent_id == resolved_agent_id

    Call uses resolved_agent_id (not raw body agent_id):
      ai_result = _lm_call_ai_provider(context, ..., agent_id=resolved_agent_id)

5. ISSUE 4 — localStorage PERSISTENCE FOR AI SELECTOR
================================================================================

  Problem:
    _lmSelectedAgentId and _lmAiMode were in-memory only; lost on page refresh.

  Fix:
    Initialization (reads localStorage at script load time):
      var _lmSelectedAgentId = (function(){
        try{ return localStorage.getItem("zyni_lm_ai_agent")||null; }catch(_){ return null; }
      })();
      var _lmAiMode = (function(){
        try{
          var m=localStorage.getItem("zyni_lm_ai_mode");
          return (m==="all"||m==="single")?m:"single";
        }catch(_){ return "single"; }
      })();

    On selector change (writes to localStorage):
      "All Agents" -> mode="all", removes zyni_lm_ai_agent
      "Auto"       -> mode="single", removes zyni_lm_ai_agent
      Agent id     -> mode="single", sets zyni_lm_ai_agent = id

    loadLiveMonitorAiProviders() validation:
      After loading agents, checks if persisted id still exists in list.
      If stale id: clears _lmSelectedAgentId + removes localStorage key.

    All localStorage calls wrapped in try/catch — unavailable env doesn't crash.

  Verification (5/5 test cases):
    [PASS] Select DeepSeek, refresh -> DeepSeek persists
    [PASS] Select All Agents, refresh -> All Agents mode persists
    [PASS] Select Auto, refresh -> Auto persists
    [PASS] Invalid mode in storage -> falls back to "single"
    [PASS] Stale agent id -> cleared after provider list reload

  Problem:
    _lm_ai_agents_config() read api_key_env from AI_AGENTS_JSON but did not
    preserve it in the returned agent dict. Provider helpers used hardcoded
    env var names, ignoring the custom api_key_env.

  Fix:
    _lm_ai_agents_config() Method A now stores internal fields:
      "_api_key_env":  key_env  (e.g. "MY_CUSTOM_KEY")

    Provider helpers updated to use priority order:
      API Key priority:
        1. os.environ.get(agent._api_key_env) if present
        2. os.environ.get(provider_hardcoded_env_name) fallback
      API Base priority:
        1. agent.api_base (already resolved from api_base_env at config time)
        2. os.environ.get(agent._api_base_env) if present
        3. provider default / env fallback

    Updated helpers:
      _lm_call_openai_compatible_agent()  ✓

    Security:
      /api/live-monitor/ai-providers uses explicit allowlist of safe fields.

      Actual key value never returned anywhere.
