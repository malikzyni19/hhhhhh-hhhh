    risks[], invalidations[], next_actions[], agent_note,
    disclaimer ("Analysis only. Not financial advice.")

  temperature: 0.2 (focused output)
  max_tokens: 900

5. AI CALL / FALLBACK BEHAVIOR
================================================================================

  _lm_call_ai_agent(context, user_message) -> dict

  Flow:
    1. Check _lm_ai_config().configured
    2. If not configured → return local fallback immediately
    3. If configured → POST to OPENROUTER_API_BASE via req.post (timeout=20s)
    4. On HTTP error → return local fallback with error note
    5. On exception → return local fallback with error note
    6. On success → parse response via _lm_parse_ai_json()

  _lm_parse_ai_json(text):
    - Strips markdown code fences if present
    - Falls back to {_parse_error, raw_text[:400]} on invalid JSON

  _lm_local_ai_fallback(context, user_message):
    - Pure rule-based analysis using health/zone/bias/session/market
    - Returns same schema as AI response
    - Verdict logic based on health_score, status, bias_aligned, zone_status
    - Generates confirmations_needed, risks, invalidations from existing facts
    - Agent note clearly states "local rule-based fallback"

6. NEW ENDPOINTS ADDED
================================================================================

  POST /api/live-monitor/items/<id>/ai-analyze
    - @login_required, ownership check, active item check
    - Builds context, calls AI agent
    - 10s cooldown: returns cached result if not forced within 10s
    - force=true in body bypasses cooldown
    - User message capped at 1000 chars
    - Stores latest_ai_analysis in snapshot_json
    - Creates ai_analysis LiveMonitorEvent (verdict, confidence, provider)
    - No MTF scan, no health recompute, no market context refresh
    - Returns {ok, item, ai_result}

  POST /api/live-monitor/items/<id>/ai-chat
    - @login_required, ownership check, active item check
    - Builds context, calls AI agent with user message
    - No DB write (chat history not stored in Phase 6)
    - No LiveMonitorEvent created
    - Message capped at 1000 chars
    - Returns {ok, reply, analysis, provider, configured}

  Total Live Monitor endpoints: 13

13. RISKS / ISSUES FOUND
================================================================================

  Context size:
    The AI context is built from compact summaries of each phase's snapshot data.
    Full raw snapshots are not sent. This keeps token usage reasonable for 900
    max_tokens. If a user's setup has very extensive MTF scan data, the context
    may still be moderately large — but compact extraction ensures manageable size.

  Provider reliability:
    OpenRouter is a third-party routing service. Any downtime falls through to
    the local rule-based fallback automatically. No crash.

  Local fallback quality:
    The rule-based fallback produces a verdict based on health_score, status,
    and bias_aligned. It is deterministic and conservative. It correctly identifies
    breach_risk, warning, and confirmed_watch states. It is not as nuanced as a
    real AI response but provides useful baseline analysis.

  Chat history not persisted:
    Phase 6 does not store chat message history in DB. Each chat call builds
    fresh context. Future Phase 7 could add conversation memory if needed.

  No model hardcoded:
    If OPENROUTER_MODEL is not set, the AI is not configured and fallback is used.
    User must set the env variable to enable real AI. The model choice is
    left to the operator — no specific model is forced.
