PHASE 11.2 — AI EXECUTION CONTEXT BUILDER
================================================================================

File:     live_monitor/ai_execution_context.py
Endpoint: GET /api/live-monitor/items/<id>/ai-execution-context?refresh=1
UI:       "AI Ctx" button inside EI card

Function: _lm_build_ai_execution_context(item, snapshot=None)

Aggregates Phase 11.1, 11.1B, Smart Entry, and Data Health into one
structured context dict used by all downstream phases.

Key output fields:
  - phase                      "phase11_2_ai_execution_context"
  - danger_context
      danger_level             blocked / high / medium / low
      danger_score             0-100
      reasons                  list of contributing factors
      blocking_reasons         list of hard-block factors
      recommended_behavior     wait / confirm / allow_touch_limit / block
      touch_limit_allowed_by_context
      confirmation_required
  - orderflow_history_quality
      overall                  high / medium / low / insufficient / none
      by_tf                    per-timeframe breakdown
      limitations              list
  - best_candidate             top-scoring execution candidate (tf, type, score)
  - smc_orderflow_fusion       zone_flow_state, confidence
  - ai_allowed_actions_preview list of currently permissible actions
  - data_health, smart_entry, execution_intelligence, mtf_orderflow_history
  - warnings, limitations

Danger scoring rules (6 rules, 0-100):
  1. Data health blocked         -> score +80, level blocked
  2. Fusion zone breaking        -> score +60, level high
  3. Candle delta divergence     -> score +25, level medium+
  4. No orderflow data           -> score +40, level high
  5. Low orderflow quality       -> score +15, level medium
  6. Fusion zone attacked        -> score +20, level medium

Orderflow quality method:
  candle_delta=high, aligned_snapshots=medium, delta_proxy=low, unavailable=none

PHASE 11.3 — AI TRADE CONTROL DECISION ENGINE
================================================================================

File:     live_monitor/ai_trade_control.py
Endpoint: GET /api/live-monitor/items/<id>/ai-trade-control?refresh=1
UI:       Rendered below AI Execution Context card (chain-fetch)

Function: _lm_build_ai_trade_control_decision(item, snapshot=None, ai_exec_ctx=None)

13-rule priority decision engine. Decision only - no execution authority.

Rule priority (highest to lowest):
   1. data_health_blocked      -> block_trade
   2. danger_blocked           -> block_trade
   3. fusion_breaking          -> block_trade
   4. danger_high              -> block_trade
   5. smart_entry_invalidated  -> pause_setup
   6. fusion_attacked          -> switch_to_confirmation
   7. danger_medium            -> switch_to_confirmation
   8. fusion_absorbing         -> switch_to_confirmation
   9. of_quality_insufficient  -> no_action
  10. of_quality_low           -> wait
  11. fusion_defended          -> allow_touch_limit
  12. danger_low               -> allow_touch_limit
  13. default                  -> no_action

Confidence scaling by orderflow quality:
  high=100%, medium=80%, low=60%, insufficient=30%, none=10%

Key output fields:
  - action           block_trade / pause_setup / switch_to_confirmation /
                     allow_touch_limit / wait / no_action
  - confidence       0-100
  - danger_level     blocked / high / medium / low
  - danger_score     0-100
  - fusion_state     breaking / attacked / absorbing / defended / neutral / unknown
  - orderflow_quality high / medium / low / insufficient / none
  - primary_reasons  list
  - supporting_reasons list
  - risk_factors     list
  - advisory_note    "Phase 11.3 generates decisions only.
                      No trade execution authority exists."

Model storage:
  LiveMonitorAIIntervention (models.py) - stores decision for audit trail.
  No automation connection. Storage only.

policy configuration. Evaluates only - never applies.

Policy modes:
  proposal_only         (default) - advisory only, no automation
  auto_testnet          - testnet automation (future, not yet active)
  future_live_execution - HARD DISABLED in Phase 11.4 (always clamped to
                          proposal_only if set)

Entry modes:
  touch_limit / confirmation / ai_controlled

5 blocking rules:
  1. data_health_gate_blocked          -> block_trade
  2. danger_level == "blocked"         -> block_trade
  3. risk_guard_status blocked         -> block_trade
  4. orderflow_quality insufficient/none -> block_trade
  5. ai_action == block_trade/pause_setup -> block/pause

Key output fields (policy_result):
  - allowed              bool
  - recommended_mode     entry mode recommendation
  - future_action        none / switch_mode / block_trade / pause_setup
  - reason               human-readable explanation
  - blocking_reasons     list
  - future_ready_state   apply_pending: false (always - never applied)
  - advisory_note        "Phase 11.4 evaluates automation policy only.
                          No execution authority exists."

SAFETY GUARD ORDER OF EXECUTION (signed requests)
================================================================================

When _lm_bt_signed_request(path, params) is called:

  Step 1  method == "GET"?                  NO -> method_not_allowed_phase11_5
  Step 2  path in _BT_SIGNED_ALLOWED_PATHS? NO -> signed_path_not_allowed_phase11_5
  Step 3  _lm_bt_is_execution_path(path)?  YES -> execution_path_blocked_phase11_5
  Step 4  _lm_bt_is_testnet_only()?         NO -> locked to testnet only
  Step 5  _lm_bt_credentials_available()?   NO -> credentials_not_configured
  Step 6  Build HMAC-SHA256 signature
  Step 7  Issue GET request to demo-fapi.binance.com

No network call is made until steps 1-5 all pass.
